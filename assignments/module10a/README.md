## Framework Context

The original assignment teaches the Laravel 12 Livewire Starter Kit. This
course-aligned implementation is mounted inside the larger CS85 portfolio,
which currently runs Laravel 13 and a custom Blade/controller authentication
layer. It preserves the assignment's required behavior and code-reading surface
without pretending that the portfolio application was generated by the
Livewire starter kit.

The advanced track adds email verification, GitHub OAuth, TOTP MFA, recovery
codes, session revocation, administrator authorization, rate limits, and audit
events. Those features extend the assignment; they do not replace the required
dashboard, secret page, concept answers, or screenshots.

## Short-Answer Questions

### 1. What is the difference between authentication and authorization?

Authentication verifies who a user is. For example, Laravel checks an email and
password before creating an authenticated session. Authorization happens after
authentication and decides what that verified user is allowed to do, such as
whether the user may open an administrator page or edit another record.

### 2. Why are passwords hashed instead of stored as plain text?

A password hash is a one-way representation of the password. Laravel stores the
hash and verifies future login attempts against it, so the original password is
not recoverable from the database. If database data is exposed, attackers do not
immediately receive every user's plain-text password.

### 3. Which file defines the `/login` and `/register` routes?

The authentication routes are defined in `routes/auth.php`. The main
`routes/web.php` file loads that route file.

### 4. What does the `auth` middleware do?

The `auth` middleware runs before a protected route. If the request has an
authenticated user, Laravel allows it to continue. If a signed-out visitor
requests `/dashboard` or `/secret`, Laravel redirects that visitor to `/login`
and remembers the intended URL so the user can continue after signing in.
